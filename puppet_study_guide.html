üìö What is Puppet? (Theory - 15 minutes)

ü§ñ Puppet Definition

Puppet is an open-source configuration management tool that automates the deployment, configuration, and management of servers and applications across your infrastructure.

üéØ Key Benefits

  • Automated configuration management
  • Consistent server states
  • Reduced manual errors
  • Scalable infrastructure

üèóÔ∏è Architecture Components

  • Puppet Master: Central server
  • Puppet Agent: Client nodes
  • Manifests: Configuration files
  • Modules: Reusable code

üîß Installation & Setup (Practice - 20 minutes)

‚ö†Ô∏è Prerequisites: Docker Desktop installed and running

üê≥ Step 1: Setup Puppet Master Container

docker run -d --name puppet-master --hostname puppet-master -p 8140:8140 puppet/puppetserver:7.4.1

This creates a Puppet Master server accessible on port 8140

üîó Step 2: Setup Puppet Agent Container

docker run -d --name puppet-agent --hostname puppet-agent --link puppet-master:puppet puppet/puppet-agent:7.4.1

This creates a Puppet Agent that can communicate with the master

‚úÖ Step 3: Verify Installation

docker exec -it puppet-master puppet --version
docker exec -it puppet-agent puppet --version

Both should show Puppet version 7.4.1

üìù Puppet Language Basics (Theory + Practice - 25 minutes)

üî§ Puppet Language Overview

Puppet uses a declarative language where you describe the desired state of your system, not how to get there.

üè† Basic Syntax: Resources

# Basic Resource Syntax resource_type { 'title': attribute => value, attribute => value, } # Example: File Resource file { '/tmp/hello.txt': content => 'Hello, Puppet World!', mode => '0644', owner => 'root', group => 'root', }

üì¶ Common Resource Types

üìÑ File Resource

file { '/etc/myapp.conf': content => template('myapp/config.erb'), mode => '0644', notify => Service['myapp'], }

üì¶ Package Resource

package { 'nginx': ensure => installed, }

‚öôÔ∏è Service Resource

service { 'nginx': ensure => running, enable => true, }

üë§ User Resource

user { 'webapp': ensure => present, home => '/home/webapp', managehome => true, }

üéØ Variables & Facts (Practice - 20 minutes)

üìä Understanding Facts

Facts are pieces of information about nodes that Puppet automatically discovers, like OS version, IP address, memory, etc.

üîç Viewing Facts

docker exec -it puppet-agent facter
docker exec -it puppet-agent facter os
docker exec -it puppet-agent facter networking.ip

üìù Using Variables

# Variable Declaration $my_variable = 'Hello World' $package_list = ['vim', 'curl', 'git'] # Using Facts as Variables $os_family = $facts['os']['family'] $ip_address = $facts['networking']['ip'] # Example Usage file { '/tmp/system-info.txt': content => "OS: ${os_family}\nIP: ${ip_address}\n", mode => '0644', }

üîó Resource Relationships (Practice - 15 minutes)

‚ö° Managing Dependencies

Puppet resources can depend on each other. You can control the order of resource application using relationships.

üîß Relationship Types

‚û°Ô∏è Before/Require

package { 'httpd': ensure => installed, before => Service['httpd'], } service { 'httpd': ensure => running, require => Package['httpd'], }

üîî Notify/Subscribe

file { '/etc/httpd/conf/httpd.conf': content => template('httpd/httpd.conf.erb'), notify => Service['httpd'], } service { 'httpd': ensure => running, subscribe => File['/etc/httpd/conf/httpd.conf'], }

üß™ Lab Exercise: Database Server with Dependencies

class database { package { 'mysql-server': ensure => installed, } file { '/etc/mysql/my.cnf': content => template('database/my.cnf.erb'), require => Package['mysql-server'], notify => Service['mysql'], } service { 'mysql': ensure => running, enable => true, require => Package['mysql-server'], subscribe => File['/etc/mysql/my.cnf'], } }

‚ö†Ô∏è Common Pitfalls to Avoid

  • Not using version control for manifests
  • Hardcoding values instead of using variables
  • Ignoring resource dependencies
  • Not testing changes in development first
  • Creating overly complex manifests

‚ú® Best Practices

  • Use modules for reusable code
  • Follow the DRY principle (Don't Repeat Yourself)
  • Use Hiera for data separation
  • Test with --noop before applying
  • Document your code with comments
  • Use version control (Git)
  • Monitor Puppet runs and reports
